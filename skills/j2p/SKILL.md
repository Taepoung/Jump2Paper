# Paper2Web 스킬

학술 논문을 하나의 자체 완결형 인터랙티브 HTML 페이지로 변환합니다.
논문의 핵심 아이디어와 스토리를 중심으로, 입문자부터 전문가까지 모두가
단계적으로 깊은 이해에 도달할 수 있는 경험을 제공합니다.

## 핵심 철학

결과물은 논문을 단순히 재포맷한 것이 **아닙니다**.
**독자가 저자의 연구 과정을 간접 체험하는 경험**입니다 — 저자가 어떤 문제에 부딪혔는지, 왜 이 방향을 선택했는지, 어떤 고민 끝에 이 수식/구조/설계에 도달했는지를 함께 따라가는 것입니다.

**스토리의 뼈대는 연구자의 고민 흐름을 따릅니다:**
> "이 문제가 왜 어려운가?" → "다들 이렇게 풀었는데 왜 안 됐나?" → "그럼 다른 방향이 있지 않을까?" → "이 아이디어가 맞는지 어떻게 확인하지?" → "이렇게 설계하면 되겠다, 하지만 이건 포기해야 한다" → "결과가 이를 뒷받침하는가?"

단순히 "논문이 무엇을 했는가"가 아니라, **"왜 이 선택을 했는가"와 "어떤 고민이 이 결과를 만들었는가"** 를 독자가 느낄 수 있어야 합니다.

**논문의 언어를 그대로 가져오세요:**

논문에서 사용하는 단어, 문장, 표현은 모두 의도적으로 선택된 것입니다 — 저자가 오랜 고민 끝에 가장 정확하다고 판단한 언어입니다. 설명을 전부 paraphrase하지 말고, 논문의 원문을 직접 인용하면서 그 옆에 해설을 붙이는 방식을 사용하세요.

예시:
> 논문 원문: *"We propose a novel attention mechanism that..."*
> → 이 문장에서 "novel"은 단순한 수식어가 아닙니다. 기존 attention이 갖는 _____ 한계를 명시적으로 극복한다는 선언입니다.

이렇게 하면 독자는 논문 원문의 언어에 익숙해지고, 논문을 직접 읽을 때도 따라갈 수 있게 됩니다. 핵심 정의, 핵심 주장, 핵심 결론 문장은 반드시 원문 인용과 함께 제시하세요.

**논문 외 내용을 덧붙일 수 있습니다:**

이해를 돕기 위한 보충 설명, 배경 지식, 비판적 평가, 추론은 적극적으로 추가하세요. 단, 논문에서 온 내용과 구분할 수 있도록 명시합니다.

- 논문 내용 → 별도 표시 없이 서술
- 보충 설명 (배경 지식, 직관 등) → `[보충]` 태그 또는 "이 개념을 이해하려면..."으로 시작
- 비판적 평가 / 추론 → `[평가]` 태그 또는 "논문은 명시하지 않지만..."으로 시작

덧붙이는 내용이 명확히 구분될 때, 독자는 논문을 신뢰하면서도 더 깊이 생각할 수 있습니다.

**목표는 계층 구분이 아닌 계층 해소입니다:**

어떤 배경을 가진 독자든, 페이지를 다 읽고 나면 **논문의 핵심을 전문가 수준으로 이해**할 수 있어야 합니다. 계층은 시작점이지 도달점이 아닙니다.
콘텐츠를 숨기거나 단순화하는 것이 아니라, **이해의 경로를 여러 개 제공**하는 것입니다.

1. 배경 지식이 없는 사람도 "이 논문이 왜 중요한지" 납득하고 읽기 시작할 수 있어야 합니다
2. 끝까지 읽은 사람은 누구든 수식·아키텍처·설계 결정을 전문가 수준으로 이해할 수 있어야 합니다
3. 논문에 있는 내용은 **하나도 빠뜨리지 않아야** 합니다
4. 저자의 고민과 선택의 이유가 느껴져야 합니다

**비유는 최소한으로, 정확하게:**

비유는 처음 개념의 문을 열어주는 용도로만 사용하세요. 비유를 과도하게 사용하거나 끝까지 유지하면 오히려 원래 개념의 정확한 의미를 가립니다. 비유를 쓴 직후에는 반드시 실제 개념으로 돌아와야 합니다.

- 비유는 "입구" 역할만: 직관을 열고 나면 즉시 실제 용어와 정의로 전환
- 부정확한 비유보다 정확한 원문이 낫습니다
- 하나의 개념에 비유를 여러 겹 쌓지 마세요

## 1단계 — HTML 초기화

콘텐츠를 쓰기 전에 HTML 파일을 먼저 만듭니다.

```bash
sed -n '9,757p' skills/j2p/assets/components.html | tr -d '\0' > paper.html
```

이때 HTML에는 CSS와 JS가 포함되어있습니다.
해당 부분은 읽을 필요가 없으므로 **앞으로의 과정에서는 <body>부분만 읽습니다.**

---

## 2단계 — 디자인 시스템 안내

**지금 `skills/j2p/references/design_system.md`를 읽으세요.**

컴포넌트가 필요한 위치에 도달하면, `skills/j2p/references/design_system.md`에서 해당 컴포넌트의 행 범위를 확인하고 Python으로 paper.html의 해당 줄에 직접 삽입합니다. 컴포넌트 내용은 컨텍스트에 들어오지 않습니다.

```bash
# python scripts/insert_component.py <start> <end> <insert_after> [target]
# 예: 수식 블록 (§2.2, 793~821행)을 paper.html 42번 줄 뒤에 삽입
python skills/j2p/scripts/insert_component.py 793 821 42
```

삽입 후 **Edit 툴로 placeholder 텍스트만 채웁니다.** 시각화 컴포넌트(§3)도 동일하게 삽입하며, HTML + CSS + JS가 자체 완결되어 있으므로 내용만 교체하면 됩니다.

## 3단계 — 논문 파싱

### 입력 유형 결정

인자와 업로드 파일을 확인해 입력 유형을 판단합니다:

| 조건 | 처리 |
|---|---|
| 파일명 인자 있음 (`.pdf`) | `python skills/j2p/scripts/parse_pdf.py filename.pdf` → `/tmp/paper_main.txt`, `/tmp/paper_appendix.txt`, `/tmp/fig_main-*.jpg`, `/tmp/fig_appendix-*.jpg` 생성 |
| 파일명 인자 있음 (`.tex`) | `main.tex` 읽기 — `\title` `\abstract` `\section` `\begin{equation}` 그림에 집중, bibliography 전까지만 |
| 인자 없음 | 앱 업로드 파일 확장자로 판단 후 위 처리 적용 |
| 인자 없음 + 업로드 없음 | 사용자에게 파일 요청 후 재시작 |

## 4단계 — 출력 언어 결정

인자로 언어가 주어지면 해당 언어로 전체 페이지를 구성하세요. 주어지지 않으면 한국어로 작성하세요.

**언어 처리 원칙:**
- 설명, 해설, UI 텍스트는 모두 지정된 언어로 작성
- 논문 고유명사(모델명, 알고리즘명, 데이터셋명 등)는 원문 그대로 유지 (번역 금지)
  - 예: "Transformer", "BERT", "ImageNet", "softmax" → 그대로
- 일반 기술 용어를 번역할 경우, 반드시 원문을 괄호로 병기
  - 예: 어텐션 메커니즘 (attention mechanism), 잔차 연결 (residual connection)
- 논문 원문 인용은 원어 그대로 유지하고, 번역이 필요하면 인용 아래에 별도로 제공

## 5단계 — 내용 작성

**`/tmp/paper_main.txt`를 읽고** 내용을 작성합니다. (LaTeX의 경우 `main.tex`)
Figure가 필요한 섹션에서는 `/tmp/fig_main-*.jpg`를 참조합니다.
추가적인 정보를 수집하면서 HTML 파일에 작성합니다.
인터랙티브 컴포넌트의 CSS/JS는 항상 별도 `<style>`/`<script>` 태그로 분리해서 추가합니다.

내용을 작성할 때 챙겨야 할 공통 항목:
- 논문 원문 인용이 필요한 곳에는 원문을 직접 가져옵니다 (`p2w-quote` 컴포넌트)
- 수식은 LaTeX 그대로 MathJax로 렌더링합니다 (`p2w-equation` 컴포넌트)
- `[보충]` / `[평가]` 태그로 논문 외 내용을 구분합니다
- 해당 내용이 논문에 없는 섹션은 삭제합니다. (내비게이션 링크도 함께 제거)
- 각 컴포넌트들은 **고유의 id를 갖고 구분 될 수 있어야함.**
  | 식별자 | id 예시 |
  |---|---|
  | SVG marker id | `id="arr-method"` |
  | SVG marker 참조 | `url(#arr-method)` |
  | 아키텍처 함수 | `p2wShowDescMethod(this)` |
  | 스텝 플레이어 변수 | `let stepsMethod`, `let curMethod` |
  | canvas id | `id="throughput-chart"` |
  | 설명 표시 div | `id="arch-desc-method"` |
- **모든 요소들은 볼 수 있도록** 설정되어야함.
- 이 시각화가 의도한 개념을 실제로 보여주는가? 텍스트 설명을 그대로 옮긴 수준이면 **다시 구현합니다**.
- 인터랙션이 실질적인 이해를 더하는가? 클릭/슬라이더/스텝이 의미 있게 작동하는가?
- 미흡하다면 **수정**해서 완성합니다.


### 섹션별 작성 가이드

#### 1. 히어로 섹션(필수)
→ 컴포넌트: `§2.1 히어로`
```
논문 제목 (크고 타이포그래픽하게, 원래 제목 유지) 
저자 · 학술지 · 연도  [arXiv 링크]

관점 문장 — 독자가 본문에 들어가기 전에 가져야 할 렌즈
  - 논문 전체를 관통하는 핵심 아이디어, 주장, 또는 관점을 한 문장으로
  - "이 논문은 ___를 ___의 관점에서 본다" 형태가 이상적
  - 요약이 아니라 관점: 읽는 내내 머릿속에 두어야 할 프레임
  - 가능하면 저자의 원문에서 가장 본질적인 문장을 인용하거나 그 정신을 살림

관련 분야 키워드 태그 (Google Scholar 링크)
```

#### 2. 문제의 배경 — 기존 연구의 한계(필수)
→ 컴포넌트: `§2.5 callout (negative)` + `§2.3 논문 인용`
- 이 분야가 지금까지 어떤 방식으로 문제를 풀어왔는지 설명
- 각 선행 연구 접근법의 **구체적 한계**를 나열
- "그래서 이 논문은 _____ 한계를 해결하려 한다"로 마무리
- 필요한 경우 비유/예시로 입구를 열되, 즉시 실제 개념으로 전환

#### 3. 이 논문의 선택 — 핵심 아이디어와 트레이드오프(필수)
→ 컴포넌트: `§2.5 callout (positive/tradeoff)` + `§3.1/§3.3` (핵심 아이디어 시각화)
- 논문이 기반으로 삼은 핵심 아이디어/통찰을 스토리로 전개
- **왜 이 접근법인가?** — 대안 대비 이 방향을 선택한 근거
- **무엇을 포기했는가?** — 트레이드오프를 명시적으로 설명
  (예: "이 방법은 A를 얻기 위해 B를 희생했습니다")
- 애니메이션/인터랙티브 다이어그램으로 핵심 개념 시각화 (SVG/Canvas)

#### 4. 방법론 — 전체 구조부터 세부까지 (표현할 수 있다면 모두 필수)
해당 부분들의 경우, 표현할 내용이 없다면, 구조에서 제거해도 상관없습니다. 
**하지만 내용이 있는 경우 무조건 작성해야합니다.**

1. **아키텍처 & 데이터 플로우**
    → 컴포넌트: `§3.1 블록 다이어그램` + `§3.2 계층형 드릴다운` + `§3.3 데이터 플로우`

    **전체 아키텍처**
    - 입력 → 모듈 → 출력 흐름을 SVG/Canvas로 그림
    - 각 모듈 **클릭 시** 드릴다운 패널 표시 (전체 → 서브모듈 → 내부 연산 순서로)
    - 모듈 간 연결선에 **데이터 shape / type** 표기
    - 연결 방식(순차 / skip / cross-attention / gate)을 선 스타일로 구분

    **내부 모듈 드릴다운 — 모듈 클릭 시 표시할 요소**
    모델 아키텍처(Transformer, CNN, GNN 등)라면 각 모듈 안에서:

    | 항목 | 표현 예시 |
    |---|---|
    | **내부 연산 순서** | LayerNorm → QKV proj → Attention → FFN 등 흐름 다이어그램 |
    | **각 연산의 역할** | 왜 이 순서인가, 대안(Pre-norm vs Post-norm 등)과 차이 |
    | **가중치 구조** | shape `(d_model, d_ff)` 명시, 초기화 방식, 레이어 간 공유 여부 |
    | **실제 연산 표현** | matmul / softmax / einsum 수식 + 시간·공간 복잡도 `O(...)` |
    | **연결 패턴** | skip connection, cross-attention, gate의 구조와 설계 이유 |
    | **차원 변환** | 각 연산 전후 tensor shape 변화 `[B,T,d] → [B,T,4d] → [B,T,d]` |
    | **수치 구체화** | d_model, num_heads, d_ff 등 논문에 나온 실제 값 |

    **데이터 플로우**
    - 레이어마다 **tensor shape 변화** 명시
    - 각 변환 단계에서: **무슨 연산**인가 / **왜 이 변환**인가 / **대안 트레이드오프**
    - "다음 단계" 버튼으로 사용자가 직접 진행 (자동 재생 금지)

2. **수식 & 알고리즘**
    → 컴포넌트: `§2.2 수식 블록` (변수 테이블 + 해설 패널 포함)
    - 논문에 등장하는 **모든 수식**을 MathJax로 렌더링 (중요도에 관계없이 전부)
    - 각 수식마다 반드시 다음을 모두 설명:
      - **왜 이 수식이 나왔는가** — 이 수식이 필요한 이유, 어떤 문제를 표현하려는가
      - **각 항/기호의 의미** — 수식을 구성하는 모든 요소를 직관적으로 해석
      - **수식의 유도 과정** — 어떤 가정이나 조건에서 이 형태가 도출되었는가
      - **수식이 변해갈 때 왜 그렇게 바뀌는가** — 다음 수식으로 넘어가는 각 변환 단계의 근거
      - **다른 형태로 바꿀 수 있는가** — 대안적 표현이 가능하다면, 그것과 어떻게 다른가
      - **이 수식이 가지는 한계 또는 가정** — 어떤 조건 하에서만 성립하는가
    - 수식 간의 **흐름 연결**: 앞 수식에서 뒤 수식으로 이어지는 논리적 흐름을 명시
    - 수식 아래에 **변수 정의 테이블 항상 표시**: 변수 / 의미 / 비고 3열 구성
    - **해설 패널 항상 표시**: 탭 없이 "직관적 해설 → 수학적 유도" 순으로 레이블과 함께 모두 노출. 독자가 선택해서 읽을 수 있게 선택권을 주되, 숨기지는 않음
    - HTML 구조는 `skills/j2p/references/design_system.md` 수식 블록 컴포넌트 참고
    - 파라미터 조작으로 결과를 직접 확인할 수 있는 수식이라면 → **§3.5 수식 슬라이더**를 수식 블록 바로 아래에 추가

3. **코드/의사코드**
  → 컴포넌트: `§3.4 스텝 플레이어` (단계별 실행 + 각 스텝에서 변수 상태 표시)
  - 핵심 알고리즘을 의사코드 또는 코드로 제시
  - 각 라인/블록의 의미와 역할 설명
  - 왜 이렇게 구현했는지, 대안과의 차이 설명

4. **구현 세부사항**

    논문마다 등장하는 공학적 문제는 다릅니다. 아래는 자주 나오는 유형의 **예시**입니다 — 논문에서 발견되는 문제를 각각 "문제 → 해결책 → 수치" 스토리로 분리해 작성하세요.

    **예시 유형과 표현 방법**

    | 공학적 문제 유형 | 확인할 내용 | 사용 컴포넌트 |
    |---|---|---|
    | **수치 안정성** | overflow/underflow 발생 위치, log-space·clipping 등 해결 방법 | 문제 수식 → 안정화 수식 비교 (`§2.2`) + `§2.5 callout` |
    | **메모리 병목** | OOM 발생 위치, tiling·checkpointing·KV cache 전략, 전후 메모리 수치 | `§3.11 행렬 시각화` (tiling) + `§3.4 스텝 플레이어` (할당 과정) + `§3.13 메트릭 카드` |
    | **속도 병목** | bottleneck 연산, 커스텀 커널(CUDA/Triton) 이유, fusion 전후 latency | `§3.8 Chart.js 차트` (연산별 시간) + `§3.13 메트릭 카드` (Before/After) |
    | **정밀도 트레이드오프** | FP16/BF16/FP32 혼용 구간, loss scaling, 정확도 변화 | 레이어별 정밀도 표 (`§2.6`) + `§2.5 callout` |
    | **병렬화 / 분산 학습** | 데이터·텐서·파이프라인 병렬 구조, gradient sync 비용, 샤딩 전략 | `§3.1 블록 다이어그램` + `§3.8 Chart.js 차트` + `§3.7 비교 슬라이더` |

    - 하이퍼파라미터: learning rate, batch size, optimizer (β1/β2/ε), weight decay, dropout, warmup steps — 논문 값 그대로 표로 정리
    - 학습 스케줄: scheduler 종류, decay 방식

## 8단계 — 완결성 및 치명적 UX 오류 점검

- [ ] **단일 HTML 완결성**: 로컬 파일 참조 없이 단일 파일로 더블클릭 실행 시, 모든 화면 요소와 CDN 기능이 정상 작동함
- [ ] **콘텐츠 누락 없음**: 논문의 수식·그림·표가 모두 포함되었으며, 아키텍처 핵심 구조와 트레이드오프 서술이 누락되지 않음
- [ ] **ID 및 변수 충돌 방지 (중요)**: 페이지 내 `id` (SVG `<marker>`, canvas 등) 중복이나 JS 요소 `null` 참조로 인한 기능 먹통 현상이 전혀 없음
- [ ] **전역 스타일 캡슐화**: 각 컴포넌트에 적용된 CSS 스타일이나 JS가 페이지의 전체 레이아웃 구성(여백, 헤딩 등)을 오염시키거나 망가뜨리지 않음
- [ ] **레이아웃 깨짐 / 이탈 방지**: 모바일 뷰(`max-width: 600px`)에서 표/수식이 가로로 잘리거나, 호버 툴팁 요소가 화면 밖으로 벗어나지 않음
- [ ] **읽기 흐름 방해 금지**: 중요한 수식·해설이 클릭 요소 뒤에 심하게 감춰지거나, 독자의 시선을 방해하는 강제 자동 애니메이션이 없음

**미통과 항목이 있으면** 해당 영역 코드를 다시 수정한 후 체크리스트를 재확인합니다. 모든 검증을 통과해야 완성으로 간주합니다.

## 9단계 — 대용량 데이터 삽입 (원본 Figure / Table)

7단계까지의 모든 내용 점검과 구조/UX 테스트가 끝난 후, HTML 용량을 크게 늘리는 **Figure 원본 렌더링을 가장 마지막 단계에서 수행합니다**. 
미리 삽입할 경우 토큰이 비대한 상태로 계속 읽게 되어 context window가 낭비되고 코드 수정 작업이 지연될 수 있습니다.

### Context-Safe 삽입 방식 (Placeholder 전략)

**절대로 거대한 Base64 문자열을 툴로 직접 붙여넣지 마십시오.** AI가 해당 내용을 다시 읽게 되어 성능을 저하시킵니다. 아래와 같이 Python을 활용해 대화 내역에 남지 않도록 삽입합니다.

1.  **Placeholder 작성**: 이미지를 넣을 공간에 `{{FIG_1_B64}}`와 같은 placeholder를 미리 적어둡니다.
    *   예: `<img src="data:image/jpeg;base64,{{FIG_1_B64}}" alt="Figure 1">`
2.  **이미지 파일 확인**: 3단계에서 추출된 `/tmp/fig_main-*.jpg` 파일 경로를 확인합니다.
3.  **Python으로 직접 주입**: 대화 내역에 Base64가 남지 않도록 파일 시스템에서 직접 치환합니다.

```bash
# python skills/j2p/scripts/inject_figure.py <image_path> <placeholder> [target]
python skills/j2p/scripts/inject_figure.py /tmp/fig_main-000.jpg {{FIG_1_B64}}
```

---

→ 컴포넌트: `§2.4 [보충]/[평가] 뱃지` + (이전/이후 비교가 있는 figure → `§3.7 비교 슬라이더`) + (단계별 흐름 figure → `§3.4 스텝 플레이어`)

- 모든 figure/table 아래 **반드시** 해설 단락을 작성합니다:
  - **논문 내 위치**: 몇 섹션, 몇 페이지에 있는가 — 독자가 원본에서 바로 찾을 수 있도록 명시
  - **무엇을 보여주는가**: 이 그림/표의 목적과 내용을 설명
  - **주목할 포인트**: 수치·패턴·비교에서 독자가 놓치지 말아야 할 것
  - **논문 주장과의 연결**: 이 그림/표가 논문의 어떤 주장을 뒷받침하는가
